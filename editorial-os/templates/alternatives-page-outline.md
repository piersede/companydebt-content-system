# Alternatives Page Template — Company Debt Editorial OS

## Required first-fold sequence (in order)

1. H1
2. Publication date + author metadata
3. Verdict block (40–60 words: who should switch, who should add a specialist tool, who should stay)
4. At-a-glance decision block (3-item bulleted quick answer, using guided scenario language — see Rule K)
5. Perspective disclosure (see §12, `11-comparison-governance.md`)
6. Verification note (what was checked, when, what evidence basis)
7. Scope note (what categories this page covers; whether replacement, augmentation, or both)
8. First intro paragraph

## Quick answer section
Heading: What are the main alternatives to [Product] for small UK accounting practices?
- 40–70 word direct answer covering: main alternatives, when to switch, when to add specialist tool
- Use short bullets, not paragraph prose

## Comparison matrix (required)
Columns: Tool | Best for | What it does best | Where it struggles | Document collection / chasing | Automation level | Integrations | Pricing model | Practice size fit

## Operational distinction section (required)
Heading: Practice management software vs client document collection tools
Must explain:
- What practice management systems do
- What document collection / chasing tools do
- Why firms confuse the two
- Why switching category does not always solve the original workflow problem

Must explain workflow mechanics, not just features:
- How reminders or tasks are triggered
- What staff must do manually
- What happens when a client responds
- What automation actually changes

## Alternatives analysis (one subsection per tool, required)
Each tool subsection must include:
- **Best for:** [one line]
- **Weak for:** [one line]
- How chasing / reminders work in practice (workflow mechanics)
- What a staff member actually has to do
- Whether client response stops the workflow automatically
- Typical pricing structure
- Best fit by practice size

## Insight block (required, 1 minimum)
Approved themes:
- Where [incumbent] starts to strain as firms grow
- What changes operationally between 100 and 200 clients
- Why adding a specialist layer can outperform full migration
- When deadline tracking is not the true bottleneck

Rules:
- Frame as editorial analysis, not product description
- Synthesise patterns across tools or practice stages
- Must not repeat a product feature list
- Must add non-obvious expert synthesis

## Decision blocks (required, minimum 2)
Format: Callout/aside block with heading + 3–5 bullet conditions

**Required patterns (guided scenario language — Rule K):**
- "If you want to replace [incumbent] entirely:"
- "If you want to keep [incumbent] but fix [specific problem]:"

**Optional adjacent retention block (separate from alternatives list — Rule L):**
- "You may not need an alternative if…"
- "When staying on [incumbent] still makes sense"

## House-product-fit block (required where house product appears)

*Source of truth: `11-comparison-governance.md` §11 (house-product containment). Fields below are a template rendering of that rule.*

Fields:
- **Best fit:** who should use it and under what conditions
- **Not a fit:** who should not use it and why
- **Required conditions:** what must be true about the practice for adoption to make sense

## Snippet-answer blocks (required, minimum 2)
Purpose: short answers to likely search queries for featured snippet eligibility
- 40–60 words per answer
- Definition paragraph or short list format
- Must answer the question directly in the first sentence

## FAQ / extractable answers (required)
Rules:
- Answer begins directly (no scene-setting opener)
- Answer is 40–60 words before any elaboration
- Use definition paragraphs, numbered lists, or short bullets
- Headings must match likely search phrasing

Required questions:
- What are the best alternatives to [Product] for UK accounting firms?
- What problem do [Product] alternatives usually solve?
- Should I switch practice management system or add a specialist tool?

## Internal linking requirements
- Minimum 4 in-body contextual links (not counting Related Articles)
- Minimum 2 concept links (practice management, document collection, client chasing)
- Minimum 2 product links to review pages if they exist
- First contextual link must appear before the article midpoint
- Maximum 1 contextual link per paragraph

## Trust modules

### Perspective disclosure (top-of-page)
Approved pattern: "This article was written by the Company Debt team and reflects our assessment of the options covered, based on [evidence basis]."
Must appear before substantive comparison begins.

### Verification note
Fields:
- What was reviewed (product documentation, pricing pages, feature testing)
- When it was checked (month and year)
- What evidence type was used (documentation review, product testing, both)

### Scope note
Fields:
- What categories this page covers
- Whether the page includes replacement paths, adjacent tools, or both

### Decision-trust module
Fields:
- What this page can help you decide
- What it cannot decide for you
- Where publisher perspective may shape the analysis
- Which readers may reasonably land elsewhere

### Author module
Include a credibility line tied to accounting workflow, practice operations, or SaaS evaluation.

### Disclosure
Preserve editorial disclosure, keep visually separate from CTA.

## CTA placement

*Source of truth: `09-voice-governance.md` §9 (comparison-page conclusion discipline).*

- Must appear after the true editorial conclusion
- Conclusion must answer the buyer's decision question first
- CTA must not interrupt the alternatives analysis or FAQ sections
- CTA must feel like the next step after the analysis, not a hard pivot into pitch
