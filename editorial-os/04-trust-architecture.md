# Trust Architecture Standard

## 1. Bylines
Use named bylines wherever possible. The page should tell the reader:
- who wrote it
- why their judgement is worth reading
- where their expertise begins and ends

The byline must not imply founder or product-builder authority unless explicitly confirmed by a human. Default byline role: editorial team member, category analyst, or informed reviewer.

## 2. Author bios
Author bios should be short, specific, and credible.

Include:
- operational experience relevant to the topic
- subject-matter relevance
- review role where relevant

Avoid:
- inflated personal branding
- vague "expert in business" language
- implied founder status

## 3. Editorial disclosures
Where relevant, disclose:
- affiliate relationships
- commercial relationships (Company Debt is our product)
- review methodology limits
- whether the product was tested directly or assessed from public evidence

On comparison pages, include: "Company Debt is our product. This review is written from that perspective. We have tried to be accurate and fair about both strengths and limitations."

## 4. Review methodology
Serious pages must include a methodology note covering:
- what sources were reviewed
- what was verified directly (product documentation, in-product testing, or both)
- what is based on editorial interpretation
- when the page was last reviewed
- what is not covered

This is not optional on decision-stage pages (reviews, comparisons, category guides).

## 5. Source presentation
Present sources in a way that increases trust, not clutter. Options:
- methodology box
- evidence notes
- labelled screenshots
- source references where appropriate

Do not cite a source you have not checked. Do not fabricate screenshots or proof assets. See `10-evidence-governance.md` for full source rules.

## 6. Updated dates
Use an updated date on serious decision pages. Do not add false freshness. If the page has not been reviewed recently, either review it or remove the freshness cue. Never add a false updated date.

## 7. Comparison table labelling and risk management

### Table metadata requirements
Every comparison table must include:
- verification date: "Comparison data last verified: [month year]"
- basis: "Based on: public documentation review / product testing / both"
- caution note: "Competitor features change frequently. Verify current capabilities before making decisions."
- scope note: what the table covers and what it does not

### Negative claim risk rule
Any negative capability claim ("No", "Not native", "Partial", "Limited", "Requires workaround") is high-risk. Each must be:
- recently verified from current product documentation (date noted)
- checked in-product (date noted)
- or flagged: "[Verify: based on documentation review, [date]]"

See `11-comparison-governance.md` for full comparison rules.

### Anti-self-serving framework rule
If a comparison table or evaluation framework is presented, it must include at least one dimension where Company Debt is not the strongest choice. See `11-comparison-governance.md`.

### Pricing transparency in comparisons
Do not criticise competitor pricing without giving actual figures. If not publicly verifiable, state: "Current pricing was not publicly available at the time of writing." If Company Debt is positioned as more economical, state the basis and limits.

## 8. Evidence-carrying claim standard
Hedging is not evidence. Any claim central to the article's argument must be:
1. Verified with a named source
2. Labelled as editorial judgement
3. Flagged for human confirmation
4. Removed

See `10-evidence-governance.md` for the full evidence governance system.

## 9. Proof signals
Strong trust signals include:
- named author with credible bio
- named reviewer
- methodology note with scope and limits
- updated date (accurate, not false)
- disclosure note (especially on comparison content)
- screenshots or proof assets (human-verified)
- clear fit / non-fit guidance
- explicit uncertainty labels
- at least one dimension where Company Debt is acknowledged as weaker (in framework content)
- specific pricing figures where cost is part of the argument
- `[HUMAN CONFIRMATION NEEDED]` flags where claims require verification

## 10. Objection stack for Company Debt pages

Resolve the top objections in the main body in this order:
1. urgency / control
2. personal exposure
3. cost / funding
4. alternatives
5. action now

FAQ may only contain lower-priority residual questions.

If affordability, HMRC visibility, or personal liability first appears in FAQ, fail trust architecture.

### Rationale

The persona framework identifies specific decision blockers for each persona-state. These must be resolved in the body where they carry structural weight, not deferred to an FAQ section where they become afterthoughts. Core objections buried in FAQ reduce decision usefulness and signal weak page architecture.

## 11. CTA discipline for distressed pages

Urgency is allowed. CTA saturation is not.

For Company Debt decision guides:
- 1 opening CTA
- 1 contextual CTA after the key route/risk section
- 1 closing CTA
- maximum 1 embedded quote/eligibility module

If multiple modules ask for the same action, collapse them into one. If "confidential" is a core trust signal, use it consistently in the chosen CTA, not repetitively across all modules.

### Fail conditions

- **FAIL** if more than 3 CTA moments appear before the closing section
- **FAIL** if overlapping CTAs with identical function appear in adjacent sections
- **FAIL** if promotional blocks are inserted between explanation paragraphs without structural separation
- **FAIL** if "free/confidential/no obligation" is repeated as standalone persuasion copy more than twice
