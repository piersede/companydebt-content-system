# Red Line Library

Hard rules that must never be broken, regardless of brief, deadline, or content type. Violation of any red line is a hard fail in the pre-publish gate.

## Voice red lines
- Never use breathless superlatives (revolutionary, game-changing, best-in-class)
- Never use vague futurist claims (the future of X, next-generation, cutting-edge)
- Never use generic SaaS cliches (seamless, end-to-end, robust, scalable solution, comprehensive platform)
- Never use inflated certainty where evidence is weak
- Never use filler intros (In today's fast-paced world, As businesses grow, In the modern era)
- Never open an article with "I think", "In my view", "I believe", or similar weak first-person scaffolding
- Never use padded evaluative language ("genuinely good", "well-executed", "useful", "robust solution", "strong offering", "compelling platform") without immediately specifying what exactly is good, for whom, and with what trade-off
- Never use filler transitions ("That said", "It is worth noting", "To be fair") as structural crutches

## Authorship red lines
- Never imply founder status, founder authority, or product-builder authority unless explicitly confirmed by a human
- Never use "I run Company Debt", "we built Company Debt", "when we started Company Debt", "our founding belief", "the problem we set out to solve", "we created Company Debt to solve this", or "I did not build Company Debt as..." unless a human has confirmed this authority
- Never use "we" to imply founding-team or product-building authority. "We" refers to the Company Debt team's editorial and operational perspective only
- Never imply personal ownership of the product or company
- Never imply first-hand product development experience

## First-person red lines
- Never use first person as filler or warmth-signalling
- Never use "I think", "I believe", "in my view", "I would say", "I find", "from what I have seen", "I want to help you understand", "I should be clear", "if I am being honest" unless the sentence passes the two-part first-person test: (a) removing it changes the meaning, not just the tone, AND (b) the claim is verifiable or human-confirmed
- Never use first person to fake product testing
- Never use first person to inflate authority beyond what is supported
- Never attribute lived experience to the author without human confirmation
- Never claim implementation experience without human-supplied evidence
- Never use first person to simulate humanity
- Never use first person in the opening sentence unless it carries a real, distinctive interpretive lens that a human editor would genuinely write

## Source honesty red lines
- Never present inferred points as verified facts
- Never present editorial judgements as objective truth
- Never flatten uncertainty to sound more authoritative
- Never omit a disclosure where one is required
- Never cite a source you have not checked or cannot attribute
- Never fabricate screenshots, proof assets, or test results
- Never use hedging language ("tend to", "often", "from what I have seen") as a substitute for evidence on claims that are central to the article's argument
- Never use decorative sourcing (citing a source to create the appearance of evidence without the source actually supporting the specific claim)

## Evidence-carrying claim red lines
- Never let a claim central to the argument rest on hedging alone. It must be verified, labelled as editorial judgement, marked for human confirmation, or removed
- Never present portal adoption rates, response-rate uplifts, time-saved claims, channel-conversion claims, client-behaviour claims, or compliance claims without a named source or an explicit `[HUMAN CONFIRMATION NEEDED]` flag
- Never treat "from what I have seen" as sufficient support for a quantitative claim
- Never treat "typically", "often", "tends to" as evidence for a load-bearing claim

## Comparison red lines
- Never present a comparison table as exhaustive without stating scope limits
- Never omit a competitor's genuine strength to favour Company Debt
- Never state competitor limitations as fact without verifiable evidence
- Never present estimated or inferred data as confirmed in a comparison table
- Never infer competitor feature absence from product origin, market focus, or company headquarters (e.g., "US-built" does not prove "no MTD audit trail")
- Always date comparison data and note verification method
- Never present a "No" about a competitor without verification from current documentation or an explicit `[VERIFY]` flag
- Never omit comparison table metadata (verification date, basis, caution note, scope note)

## Framework red lines
- Never present a buyer framework, comparison framework, or category framework where Company Debt wins on every dimension. Include at least one meaningful dimension where Company Debt is weaker
- Never design an evaluation framework that leads to a predetermined conclusion without acknowledging the bias
- Never select evaluation criteria solely to favour Company Debt

## Company Debt mention red lines
- Never introduce Company Debt as an abrupt product pivot after neutral competitor or category analysis
- Never mention Company Debt without also stating what it does not do
- Never present Company Debt as a replacement for tools that serve a different purpose without stating the limits
- Never omit the Company Debt disclosure wherever Company Debt is mentioned, in any article type
- Never end a non-Company Debt article on Company Debt product features or soft CTAs

## Pricing red lines
- Never criticise competitor pricing without giving actual figures or stating "current pricing could not be verified"
- Never position Company Debt as more economical without stating the basis and limits of the comparison
- Never omit Company Debt pricing when the article recommends cost comparisons
- Never use vague pricing language ("steep", "significant commitment", "premium") as the only price signal when the reader needs figures to decide

## Decision guidance red lines
- Never make a recommendation without supporting reasoning
- Never present buyer guidance without stating who it applies to
- Never hide trade-offs that a reader would need to make a good decision
- Never push a reader toward Company Debt without acknowledging genuine alternatives

## Customer and outcome red lines
- Never claim customer results without human-supplied evidence
- Never imply satisfaction, ease, or speed from assumptions
- Never fabricate testimonials or case study details
- Never attribute quotes without verification

## Style and readability red lines
Do not:
- overuse em dashes (maximum 1 per article, ideally zero)
- use bolding or italics as decorative emphasis
- write long wall-of-text paragraphs (3 lines maximum)
- stack multiple ideas into one dense paragraph
- format articles in a way that feels typographically busy or AI-styled
- use lists as a crutch to avoid writing proper paragraphs

## Content maintenance red lines
- Never add a false updated date
- Never present stale data as current without flagging it
- Never remove a methodology note to make a page look cleaner

## Structural red lines
- Never publish a decision-stage page without a methodology note
- Never publish a comparison page without table labelling and scope notes
- Never publish a review without a disclosure statement
- Never publish content with a score below 3 in any rubric category
- Never publish content that fails any check in the pre-publish gate (see `16-pre-publish-gate.md`)

## Compliance red lines
- Never make compliance claims (GDPR, UK GDPR, data residency, consent) without verification from primary sources or a `[HUMAN CONFIRMATION NEEDED]` flag
- Never state regulatory requirements as fact without checking current regulatory guidance
- Never imply that using a specific tool automatically satisfies compliance obligations
